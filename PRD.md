- **문서 상태**: `업데이트 (오류 수정)`

## 1. 프로덕트 개요 (Overview)

### 1.1. 한 줄 정의

기획자가 실제 개발에 사용될 디자인 컴포넌트를 조립하여, 코딩이나 복잡한 디자인 툴 없이 실물에 가까운 기획서를 작성할 수 있도록 돕는 노코드(No-code) 프로덕트

### 1.2. 문제 정의 (Problem Statement)

현재 기획자들은 다음과 같은 심각한 비효율을 겪고 있습니다.

- **괴리감과 재작업**: PPT, Keynote, Balsamiq 등으로 작성된 기획서는 시각적/기능적으로 실제 개발 결과물과 큰 차이를 보입니다. 이로 인해 개발/디자인 단계에서 의도를 재해석하고 다시 질문하는 과정이 반복되며, 불필요한 커뮤니케이션 비용과 재작업이 발생합니다.
- **높은 학습 허들**: 최종 결과물과 비슷한 기획서를 만들기 위해 Figma나 코딩을 배우는 것은 기획자에게 과도한 학습 부담을 주며, 이는 논리 설계와 정책 정의라는 본연의 업무에 집중하지 못하게 만듭니다.
- **정적인 명세의 한계**: 텍스트와 이미지 기반의 기획서는 '버튼 클릭 시 모달 노출', '토글 시 영역 확장' 등 동적인 사용자 인터랙션과 다양한 UI 상태(Error, Disabled, Loading)를 명확하게 표현하기 어렵습니다.

### 1.3. 해결 방안 (Our Solution)

우리는 기획자에게 개발/디자인팀과 동일한 '언어(Language)'를 제공하여 이 문제를 해결합니다.

- **컴포넌트 기반 설계**: 개발팀이 사용하는 실제 디자인 시스템의 컴포넌트(Button, Input, Modal 등)를 라이브러리 형태로 제공합니다.
- **직관적인 빌더**: 기획자는 코딩이나 복잡한 디자인 툴 없이, 컴포넌트를 캔버스에 끌어다 놓는(Drag & Drop) 직관적인 방식으로 화면을 설계합니다.
- **살아있는 기획서**: 기획서 자체가 실제 컴포넌트로 구성되어 있으므로, 최종 개발 결과물과의 싱크로율이 극대화되고, 개발자와 디자이너는 기획 의도를 명확하게 이해할 수 있습니다.

## 2. 목표 및 성공 지표 (Goals & Metrics)

- **비즈니스 목표**: 기획-디자인-개발 사이의 커뮤니케이션 비용을 30% 절감한다.
- **제품 목표 (MVP)**: 기획자가 하나의 완성된 사용자 플로우(3-5개 화면)를 담은 기획서를 작성하고 팀에 공유할 수 있게 한다.
- **핵심 성공 지표 (KPIs)**:
    - **활성화**: 주간 활성 사용자(WAU) 수
    - **리텐션**: 기획서 생성 후 7일 이내 재방문율
    - **핵심 행동**: 생성된 기획서의 평균 화면 수, 공유 기능 사용 횟수

## 4. 기능 명세 및 우선순위 (Features & Priority)

MVP(Minimum Viable Product, 최소 기능 제품)는 기획자가 핵심 가치를 경험하는 데 집중합니다. 기능은 크게 **'핵심 작성 경험'**과 **'협업 및 공유'**라는 두 가지 테마로 그룹화됩니다.

### 화면 빌더 인터페이스 레이아웃
화면 빌더는 크게 4개의 영역으로 구성됩니다. 이는 [bricksbuilder.io](https://bricksbuilder.io)와 같은 전문 빌더의 직관적인 UX를 차용하되, 기획자에게 맞게 단순화된 구조입니다.

- **A. 상단 바 (Top Bar)**: 프로젝트 이름, 저장 상태, 미리보기, 공유 등 전체 문서에 영향을 주는 최상위 기능 영역.
- **B. 좌측 패널 (Left Panel)**: 화면의 '구조'를 보거나 새로운 '컴포넌트'를 추가하는 영역.
- **C. 중앙 캔버스 (Center Canvas)**: 실제 화면이 조립되는 시각적인 작업 공간. WYSIWYG(What You See Is What You Get)을 제공.
- **D. 우측 패널 (Right Panel)**: 캔버스에서 선택된 컴포넌트의 '속성'을 편집하는 영역.

### 테마 1: 핵심 기획서 작성 경험 (Core Authoring Experience)

#### Epic 1: 프로젝트 및 화면 관리 (Priority: `Must`)
*기획자는 여러 기획 문서를 체계적으로 관리하고, 문서 내 화면들의 구조를 쉽게 파악할 수 있다.*

| 사용자 스토리 | 완료 조건 (Acceptance Criteria) |
| :--- | :--- |
| **1.1. 프로젝트 생성**:<br/>기획자로서, 나는 새로운 프로젝트를 생성할 수 있다. | - '새 프로젝트 만들기' 버튼이 존재한다.<br/>- 버튼 클릭 시 프로젝트 이름을 입력하는 모달이 나타난다.<br/>- 이름 입력 후 확인 시, 비어있는 새 프로젝트 편집 화면으로 이동한다. |
| **1.2. 프로젝트 목록 조회**:<br/>기획자로서, 나는 내가 생성한 프로젝트 목록을 한눈에 볼 수 있다. | - 서비스 진입 시 대시보드에서 내 프로젝트 목록이 카드 형태로 표시된다.<br/>- 각 카드에는 프로젝트 이름과 마지막 수정일이 표시된다.<br/>- 카드를 클릭하면 해당 프로젝트의 편집 화면으로 이동한다. |
| **1.3. 화면 추가/삭제/순서 변경**:<br/>기획자로서, 나는 프로젝트 내에서 화면(Screen)을 자유롭게 추가, 삭제, 순서 변경할 수 있다. | - 편집 화면 좌측 사이드바에 화면 목록이 표시된다.<br/>- '화면 추가' 버튼 클릭 시, 목록 하단에 새 화면이 추가된다.<br/>- 각 화면 항목에 마우스를 올리면 삭제 아이콘이 나타나며, 클릭 시 확인 후 삭제된다.<br/>- 화면 항목을 드래그하여 목록 내 순서를 변경할 수 있다. |

#### Epic 2: 컴포넌트 기반 화면 빌드 (Priority: `Must`)
*기획자는 제공되는 컴포넌트를 이용해 코딩 없이 실제와 유사한 화면을 시각적으로 구성할 수 있다.*

| 사용자 스토리 | 상세 시나리오 및 완료 조건 (Detailed Scenarios & Acceptance Criteria) |
| :--- | :--- |
| **2.1. 컴포넌트 탐색 및 추가**:<br/>기획자로서, 나는 필요한 컴포넌트를 쉽게 찾아 화면에 추가할 수 있다. | **[시나리오]**<br/>1. 좌측 패널 상단의 `[+]` 컴포넌트 추가 탭을 클릭한다.<br/>2. 패널에 카테고리별(Layout, Basic, Form 등)로 정리된 컴포넌트 목록이 나타난다.<br/>3. 검색창에 '버튼'을 입력해 버튼 컴포넌트를 찾는다.<br/>4. '기본 버튼' 컴포넌트를 중앙 캔버스의 원하는 위치로 드래그 앤 드롭한다.<br/><br/>**[완료 조건]**<br/>- 드롭한 위치에 버튼 컴포넌트가 시각적으로 생성되어야 한다.<br/>- 좌측 패널을 구조 탭으로 전환하면, 방금 추가한 버튼이 구조 트리에 표시되어야 한다. |

| **2.3. 컴포넌트 속성 편집**:<br/>기획자로서, 나는 화면에 배치된 컴포넌트의 내용(텍스트 등)과 주요 정책을 정의할 수 있다. | **[시나리오]**<br/>1. 중앙 캔버스에서 '기본 버튼' 컴포넌트를 클릭한다.<br/>2. 우측 패널에 해당 버튼의 속성을 편집할 수 있는 '속성 편집기'가 나타난다.<br/>3. **[컨텐츠 탭]**: `Label` 항목의 텍스트를 '확인'에서 '회원가입'으로 변경한다.<br/>4. **[정책/기능 탭]**: `비활성화 조건` 항목에 '이메일 형식이 아닐 경우'라고 정책을 텍스트로 기술한다.<br/><br/>**[완료 조건]**<br/>- 캔버스에서 컴포넌트를 선택하면, 우측 패널에 해당 컴포넌트의 속성만 표시되어야 한다.<br/>- 속성 편집기에서 값을 변경하면 중앙 캔버스에 실시간으로 반영되어야 한다. (Live-update)<br/>- **(MVP)** 편집 가능한 속성은 텍스트, 플레이스홀더, 상태(Disabled, Error) 등 기획 명세에 필수적인 항목으로 제한한다. (디자인 스타일링 기능은 최소화) |
